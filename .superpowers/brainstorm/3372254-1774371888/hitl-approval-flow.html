Section 2: HITL Router & Approval Flow

Channel-agnostic approval system supporting Cortex UI and Telegram

Approval Request Lifecycle

End-to-End Flow — From DAG Pause to Resume
PHASE 1: PAUSE
1. evaluate_dag() encounters node with approval_required: true
2. Node marked 'waiting_approval' in workflow_nodes table
3. Workflow run status updated to 'paused'
4. Create approval_request row in Supabase:
   {id, workflow_run_id, node_id, approval_type, payload, status: 'pending',
    created_at, expires_at, approved_by, approved_via, resolved_at}
5. HITL Router dispatches to all configured channels:

   → Cortex UI (SSE):
     Fire event: {type: 'approval_requested', approval_id, workflow_run_id,
                  node_id, approval_type, payload}
     UI renders rich approval view (plan diff, code preview, etc.)
     Approve/Reject buttons call REST: POST /api/workflows/approvals/{id}/resolve

   → Telegram (Bot API):
     Send message to project's chat_id:
       "🔍 Approval needed: [workflow_name] — [node_name]"
       "[summary — first 500 chars of payload]"
       "View full details: https://cortex.local/workflows/{run_id}/approvals/{id}"
       InlineKeyboard: [✅ Approve] [❌ Reject]
     Telegram callback_query handler calls same REST endpoint

PHASE 2: WAIT
6. Workflow run is suspended — no further nodes evaluated
7. Approval request has configurable TTL (default: 24h)
8. If TTL expires → auto-reject, mark node 'failed', resume DAG evaluation
9. Periodic heartbeat SSE keeps UI updated on wait duration

PHASE 3: RESUME
10. User approves via ANY channel → REST endpoint:
    POST /api/workflows/approvals/{id}/resolve
    Body: {decision: 'approved'|'rejected', comment?: string, channel: 'ui'|'telegram'}
11. approval_request updated: status='resolved', approved_by, approved_via, resolved_at
12. IF approved:
    • Node transitions: waiting_approval → queued
    • evaluate_dag() re-invoked — resumes topological walk
    • SSE fired to all channels: {type: 'approval_resolved', decision: 'approved'}
    • Telegram: edit original message to show "✅ Approved by [user] via [channel]"
13. IF rejected:
    • Node transitions: waiting_approval → failed (with rejection reason)
    • evaluate_dag() re-invoked — downstream nodes evaluated per trigger_rules
    • SSE + Telegram notified of rejection

Approval Types

What Can Be Approved
plan_review
Review an implementation plan before execution begins
UI payload: Full plan markdown, file tree, estimated scope
Telegram: Summary + link to full plan in UI
pr_review
Review changes before creating a pull request
UI payload: Diff view, commit list, PR description preview
Telegram: Stats (files changed, insertions/deletions) + link
deploy_gate
Approve before deploying or merging to protected branch
UI payload: Environment details, rollback plan, test results
Telegram: Environment + test pass/fail summary + link
custom
User-defined approval gate with custom payload
UI payload: Rendered from node's output (markdown/JSON)
Telegram: Node output summary + link

HITL Router Architecture

Channel-Agnostic Dispatch
HITLRouter
python/src/server/services/workflow/hitl_router.py
dispatches to →
UIChannel
Fires SSE event to connected browsers. Renders rich approval payload.
TelegramChannel
Sends bot message via Telegram Bot API. Handles callback_query for inline buttons.
Future: SlackChannel
Same pattern — implement ApprovalChannel interface.
Interface contract: 
class ApprovalChannel(Protocol):
    async def send_approval_request(
        self,
        approval: ApprovalRequest,
        project: Project,
    ) -> None: ...

    async def notify_resolution(
        self,
        approval: ApprovalRequest,
        decision: str,
        resolved_by: str,
    ) -> None: ...
Key design choice: All channels converge on the same REST endpoint for resolution. The HITL Router doesn't handle responses — it only dispatches. Resolution is always: POST /api/workflows/approvals/{id}/resolve This makes adding channels trivial — you only implement the send side.

Telegram Integration Detail

How Cortex Talks to the Per-Project Telegram Bot
Sending Approval Requests
1. TelegramChannel looks up project's telegram_chat_id from Supabase
2. Builds message: summary + inline keyboard buttons
3. Calls Telegram Bot API: sendMessage with reply_markup
4. Stores Telegram message_id in approval_request metadata
5. On resolution: calls editMessageText to update the original message
Receiving Approvals
1. User taps inline button in Telegram
2. Telegram sends callback_query to bot
3. Bot extracts approval_id + decision from callback data
4. Bot calls Cortex REST: POST /api/workflows/approvals/{id}/resolve
5. Cortex processes approval, fires SSE, resumes DAG
6. Bot answers callback_query with confirmation toast
CRITICAL: Where does the Telegram bot run?
The remote-coding-agent already runs per-project Telegram bots. Two options:

Option 1 (Recommended): Cortex sends approval requests to the remote-agent via REST, which forwards them through its existing Telegram adapter. The remote-agent's callback_query handler calls Cortex's approval REST endpoint. No new Telegram bot needed in Cortex.

Option 2: Cortex runs its own Telegram bot for approvals only. Separate from the remote-agent's per-project bots. More infrastructure to manage.
