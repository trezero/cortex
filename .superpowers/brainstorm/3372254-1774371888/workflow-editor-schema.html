Section 4: Workflow Editor & YAML Schema

UI-driven workflow authoring with YAML-native storage, compatible with remote-agent format

Unified YAML Schema

Extended WorkflowDefinition — Cortex additions in green
name: implement-feature
description: "Full feature implementation with plan review gate"
provider: claude
model: sonnet

# ── Remote-agent compatible fields ──
nodes:
  - id: create-branch
    command: create-branch
    context: fresh

  - id: planning
    command: planning
    depends_on: [create-branch]

  - id: plan-review                          # ← HITL gate node
    prompt: "Summarize the plan for approval"
    depends_on: [planning]
    approval:                                  # ← NEW: Cortex extension
      required: true
      type: plan_review
      ttl_hours: 24
      channels: [ui, telegram]

  - id: execute
    command: execute
    depends_on: [plan-review]

  - id: commit
    command: commit
    depends_on: [execute]

  - id: create-pr
    command: create-pr
    depends_on: [commit]

  - id: pr-review                             # ← Second HITL gate
    prompt: "Summarize changes for PR review"
    depends_on: [create-pr]
    approval:
      required: true
      type: pr_review
      ttl_hours: 48

  # ── Conditional branching (same syntax as remote-agent) ──
  - id: classify-issue
    prompt: "Classify this as BUG or FEATURE. Output JSON: {type: '...'}"
    output_format: {type: "object", properties: {type: {type: "string"}}}

  - id: hotfix-path
    command: hotfix
    depends_on: [classify-issue]
    when: "$classify-issue.output.type == 'BUG'"

  - id: feature-path
    command: planning
    depends_on: [classify-issue]
    when: "$classify-issue.output.type == 'FEATURE'"

# ── Cortex-only metadata (ignored by remote-agent) ──
cortex:
  project_id: proj_xyz                       # Scope to Cortex project
  tags: [feature, full-pipeline]
  icon: rocket                               # UI display icon
  suggested_by: pattern_discovery            # Origin tracking
Compatibility strategy: The approval: block and cortex: metadata are Cortex extensions. The remote-agent's YAML parser ignores unknown fields, so the same file works in both systems — the remote-agent simply skips the approval gates and runs straight through. This means exported workflows are immediately usable in both contexts.

Supabase Storage Model

workflow_definitions table
Column Type Notes
id uuid PK
name text Unique per project scope (null = global)
description text Human-readable description
project_id uuid | null FK → cortex_projects. Null = global workflow
yaml_content text Full YAML source (canonical storage format)
parsed_definition jsonb Parsed YAML for fast queries (auto-generated)
version integer Auto-incrementing per name+project scope
is_latest boolean Only one version is latest per name+project
tags text[] Searchable tags for filtering in UI
origin text user_created | imported | pattern_discovery
created_at timestamptz
deleted_at timestamptz | null Soft delete

Workflow Editor Page — Conceptual Layout
Form Panel (left side)
Workflow Metadata
Name, description, provider, model, tags
Node List
Sortable list of nodes. Each node expandable:
• ID, type (command/prompt/bash)
• depends_on (multi-select from other nodes)
• when: condition (text input with syntax help)
• trigger_rule (dropdown)
• approval gate toggle + config
• model/provider overrides
[+ Add Node] button at bottom
Actions
[Save] [Validate YAML] [Export .yaml] [Run Workflow]

Editable — changes sync back to form.
Bidirectional: form ↔ YAML always in sync.
Syntax highlighting + validation errors inline.
Import flow: Users can paste or upload a YAML file from .cortex/workflows/ → parsed into form fields → saved to Supabase. Export flow: Download button generates a clean YAML file (strips cortex: metadata block) compatible with remote-agent's .cortex/workflows/ directory.
