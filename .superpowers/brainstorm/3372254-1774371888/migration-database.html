Section 6: Database Schema & Migration from Current System

New tables, migration path from agent work orders, and deprecation plan

Complete Database Schema — New Tables

8 New Supabase Tables (migrations 027–034)
027: workflow_definitions
YAML-native workflow storage with versioning.
id, name, description, project_id (FK), yaml_content, parsed_definition (jsonb), version, is_latest, tags (text[]), origin, created_at, deleted_at
UNIQUE(name, project_id, version)
028: workflow_commands
DB-backed command templates (replaces .md files).
id, name, description, prompt_template (text), variables (jsonb), version, is_latest, project_id (FK nullable), created_at, deleted_at
UNIQUE(name, project_id, version)
029: workflow_runs
Workflow execution instances.
id, definition_id (FK), project_id (FK), status (pending|running|paused|completed|failed|cancelled), triggered_by (text), trigger_context (jsonb), backend_id (FK nullable), started_at, completed_at, created_at
INDEX on (status), (project_id, status)
030: workflow_nodes
Per-node state within a workflow run.
id, workflow_run_id (FK), node_id (text — matches YAML), state (pending|queued|running|waiting_approval|completed|failed|skipped|cancelled), claimed_by_backend_id (FK nullable), output (text), error (text nullable), session_id (text nullable), started_at, completed_at
INDEX on (workflow_run_id, state)
UNIQUE(workflow_run_id, node_id)
031: approval_requests
HITL approval state.
id, workflow_run_id (FK), node_id (text), approval_type (text), payload (jsonb), status (pending|approved|rejected|expired), channels_notified (text[]), resolved_by (text nullable), resolved_via (text nullable), resolved_comment (text nullable), telegram_message_id (text nullable), expires_at, created_at, resolved_at
INDEX on (status), (workflow_run_id)
032: execution_backends
Registered execution backend instances.
id, name, base_url, auth_token_hash (text), capabilities (text[]), supported_node_types (text[]), max_concurrent_nodes (int), status (healthy|unhealthy|disconnected), last_heartbeat_at, registered_at
UNIQUE(name)
033: activity_events
Unified activity stream for pattern discovery.
id, event_type (commit|conversation|workflow_run), project_id (FK nullable), repo_url (text), intent_summary (text nullable), intent_embedding (vector(1536) nullable), metadata (jsonb), created_at
INDEX on (event_type, created_at)
ivfflat INDEX on intent_embedding
034: discovered_patterns
AI-discovered workflow suggestions.
id, pattern_name, description, cluster_embedding (vector(1536)), source_event_ids (uuid[]), repos_involved (text[]), frequency_score (float), cross_repo_score (float), final_score (float), suggested_yaml (text), status (pending_review|accepted|dismissed|expired), accepted_workflow_id (FK nullable), discovered_at
INDEX on (status, final_score DESC)

Migration from Agent Work Orders

Deprecation & Transition Plan
Phase 1: Build Alongside
• New orchestration engine built as separate service module
• Agent work orders continues to run unchanged
• New API endpoints under /api/workflows/ prefix
• Existing /api/agent-work-orders/ untouched
• Convert the 6 hardcoded steps into a YAML workflow definition
• Both systems operational simultaneously
Phase 2: Bridge & Verify
• Cortex local executor wraps existing AgentCLIExecutor
• Register as execution backend to new DAG engine
• Run same workflows through both systems, compare results
• Add HITL gates to the converted workflow
• UI updated with new Workflows page
• Existing work order UI still accessible
Phase 3: Deprecate & Remove
• All work order creation routed through new engine
• /api/agent-work-orders/ endpoints return deprecation warning
• Agent work orders module removed
• Worktree/sandbox code reused by local executor
• Structured logging patterns preserved
• In-memory state replaced by Supabase persistence
What gets preserved from agent work orders:
✓ Git worktree sandbox pattern
✓ Port allocation logic
✓ Claude CLI executor (wrapped by local backend)
✓ Structured logging patterns
✓ Command .md file loading (as fallback)
✓ GitHub integration (gh CLI operations)
What gets replaced:
✗ Linear WorkflowOrchestrator → DAG engine
✗ In-memory state → Supabase persistence
✗ Hardcoded command_map → YAML definitions
✗ Separate microservice → integrated into Cortex server
✗ /api/agent-work-orders/ → /api/workflows/
✗ WorkflowStep enum → dynamic node definitions

Remote-Agent Integration Bridge

How the remote-coding-agent connects as an execution backend
New module: packages/core/src/cortex-bridge/ in remote-coding-agent repo Responsibility: Registers with Cortex, connects SSE, claims + executes nodes, reports results Node execution: Routes to existing workflow engine — DAG nodes → remote-agent's executeDagWorkflow() for local sub-DAGs, or single AI steps via sendQuery() Telegram bridge: When Cortex sends approval request → bridge forwards to the project's Telegram adapter → user approves → bridge calls Cortex's resolve endpoint Session continuity: Node results include session_id from Claude Agent SDK → stored in Cortex's workflow_nodes → passed back as context to dependent nodes Isolation: Reuses existing IsolationResolver for worktree lifecycle — Cortex doesn't need to know about isolation details
Minimal changes to remote-agent: The bridge is an additive module. The remote-agent's existing orchestrator, adapters, and workflow engine remain untouched. The bridge sits alongside them as an alternative entry point for Cortex-orchestrated work. Users can still use the remote-agent standalone without Cortex.
