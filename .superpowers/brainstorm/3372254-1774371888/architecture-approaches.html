Workflows 2.0 — Architectural Approaches

Three approaches to building the orchestration protocol between Cortex and execution backends

A

REST Gateway — Stateless Bridge

Cortex exposes a Workflow Execution API. The remote-agent (and future backends) call Cortex to register workflows, report step completions, and poll for HITL approvals.

Pros

  • Simple — standard REST, no new infrastructure
  • Backends are fully independent processes
  • Easy to add new execution backends
  • Cortex DB is single source of truth

Cons

  • Polling-based HITL adds latency (seconds)
  • Backend must handle its own retry/resume on network failures
  • No real-time state sync — UI polls too

Flow: Backend starts step → reports completion to Cortex API → Cortex evaluates DAG → if HITL needed: pauses, notifies UI + Telegram → user approves via either channel → Cortex marks approved → backend polls and resumes

B

Event Bus — SSE + Webhooks

Cortex pushes DAG state changes to backends via SSE streams. Backends push step results via webhooks. HITL approvals are pushed to the waiting backend instantly.

Pros

  • Real-time — no polling latency for HITL or UI
  • Push-based: backends react immediately to approvals
  • SSE reuses existing Cortex pattern (progress tracking)
  • Clean event-driven architecture

Cons

  • SSE connections must be kept alive (reconnection logic)
  • Need webhook registration + delivery guarantees
  • More complex than REST — debugging harder
  • Must handle SSE disconnection during HITL wait

Flow: Backend connects SSE → Cortex pushes "run node X" → Backend executes, webhooks result → Cortex evaluates DAG → if HITL: pushes pause event to UI SSE + Telegram → user approves → Cortex pushes "resume" to backend SSE

C

Hybrid — REST + SSE Notifications

REST for control plane (start workflow, report results, approve). SSE for notifications (state changes, approval requests, resume signals). Best of both worlds.

Pros

  • REST for reliability — every state change is an explicit API call
  • SSE for speed — instant notifications, no polling delay
  • Graceful degradation: if SSE disconnects, backend can poll REST
  • Matches Cortex's existing architecture (REST + smart polling)
  • UI gets real-time updates via same SSE pattern

Cons

  • Two communication channels to maintain
  • Must ensure REST and SSE stay consistent

Flow: Backend calls REST to start workflow → Cortex evaluates DAG, returns next nodes → Backend executes, calls REST with result → if HITL: Cortex persists pause state (REST), fires SSE to UI + sends Telegram notification → user approves via REST endpoint → Cortex fires SSE "resume" to backend → backend calls REST to get next nodes

⭐ Recommended — aligns with Cortex's existing patterns and provides both reliability and real-time UX
