Section 3: Orchestration Protocol & Execution Backend API

The REST + SSE contract between Cortex and execution backends

Backend Registration & Lifecycle

How Execution Backends Connect to Cortex
REGISTRATION (one-time)
1. Backend calls: POST /api/workflows/backends/register
   Body: {
     name: "remote-coding-agent",
     base_url: "http://remote-agent:3000",
     capabilities: ["claude-sdk", "codex", "worktree-isolation", "telegram"],
     supported_node_types: ["command", "prompt", "bash"],
     max_concurrent_nodes: 5,
     heartbeat_interval_seconds: 30
   }
2. Cortex stores in execution_backends table, returns backend_id + auth_token
3. Backend stores token for subsequent API calls

SSE CONNECTION (persistent)
4. Backend connects: GET /api/workflows/backends/{id}/events
   Headers: Authorization: Bearer {auth_token}
5. Cortex streams events:
   • node_ready     — "execute this node" (includes full node definition + context)
   • workflow_cancel — "abort all nodes for this workflow run"
   • approval_resolved — "HITL gate passed, resume downstream nodes"
   • heartbeat      — keep-alive ping every 15s

HEARTBEAT (ongoing)
6. Backend sends: POST /api/workflows/backends/{id}/heartbeat
   Every 30 seconds while connected
7. If heartbeat missed for 3 intervals (90s):
   • Cortex marks backend 'unhealthy'
   • Running nodes on that backend marked 'failed' with reason 'backend_timeout'
   • evaluate_dag() re-invoked to process downstream effects

Node Execution Protocol

Step-by-Step: Backend Executes a Node
Execution Backend
① Receives node_ready via SSE
Contains: node definition, upstream outputs, workflow context, working_dir
③ Calls REST to claim node:
POST /api/workflows/nodes/{id}/claim
Prevents double-execution if multiple backends
④ Executes node (Claude SDK / CLI / bash)
Streams progress updates periodically
⑤ Reports result:
POST /api/workflows/nodes/{id}/result
Body: {state, output, error?, session_id?, artifacts?}

Cortex DAG Engine
① evaluate_dag() marks node 'queued'
Fires node_ready SSE to connected backends
③ Validates claim (atomic DB update)
Returns 200 if claimed, 409 if already claimed
④ Updates node progress in DB
Fires SSE to UI for real-time progress display
⑤ Stores result in node_outputs
Calls evaluate_dag() → queues next nodes

REST API Surface

Orchestration API Endpoints
Workflow Management (UI + MCP clients)
POST /api/workflows Create workflow run from definition
GET /api/workflows/{run_id} Get workflow run status + all node states
GET /api/workflows/{run_id}/events SSE stream for UI (node state changes, progress, approvals)
POST /api/workflows/{run_id}/cancel Cancel workflow run (marks all pending nodes cancelled)
GET /api/workflows/definitions List/search workflow definitions from Supabase
Approval Management (UI + Telegram callback)
GET /api/workflows/approvals List pending approvals (filterable by project)
GET /api/workflows/approvals/{id} Get full approval payload (plan, diff, etc.)
POST /api/workflows/approvals/{id}/resolve Approve or reject (from any channel)
Backend Protocol (execution backends only)
POST /api/workflows/backends/register Register new execution backend
GET /api/workflows/backends/{id}/events SSE stream for backend (node_ready, cancel, resume)
POST /api/workflows/backends/{id}/heartbeat Backend heartbeat (every 30s)
POST /api/workflows/nodes/{id}/claim Atomically claim a queued node for execution
POST /api/workflows/nodes/{id}/progress Stream progress update during execution
POST /api/workflows/nodes/{id}/result Report node completion/failure + output
Workflow Definitions (UI editor)
GET /api/workflows/definitions List all definitions (with search/filter)
POST /api/workflows/definitions Create new definition (YAML stored in Supabase)
PUT /api/workflows/definitions/{id} Update definition (versioned — old version preserved)
DELETE /api/workflows/definitions/{id} Soft-delete definition
POST /api/workflows/definitions/{id}/export Export as YAML file (for .cortex/workflows/)
